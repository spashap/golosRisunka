## 0. Принципы работы с этим документом

1. **Простота — закон.** На каждой развилке реализации выбирай простейший рабочий вариант. Не добавляй абстракций, очередей, кэшей, микросервисов «на вырост». Один сервер, один процесс приложения, SQLite, файлы на диске.
2. **Гибкость — закон.** Всё, что увидит пользователь (тексты, цены, поля формы, цвета, структура отчёта), будет меняться после запуска. Поэтому: тексты и цены — в конфигах/шаблонах, не в коде. Поля формы — конфигурируемый список. Цвета — CSS-переменные.
3. **Переиспользование — закон (требование заказчика, обязательное).** Глобальная дизайн-система: один набор CSS-переменных и один набор базовых компонентов (кнопка, карточка, полароид, бейдж, бар оценки, секция). Каждый блок каждой страницы собирается из них. Изменение в одном месте меняет весь сайт. Запрещено создавать новые одноразовые стили для каждого блока/страницы.
4. **Свобода реализации.** Этот документ фиксирует ЧТО и ПОЧЕМУ. КАК — на усмотрение разработчика (Claude Code), если не нарушены пункты 1–3 и явные решения ниже. Конкретные имена таблиц, структура каталогов, выбор Flask vs FastAPI — свободные решения.
5. **Дизайн — placeholder.** Дизайн-направление зафиксировано (раздел 4), но финальная отделка будет итерироваться после запуска. Не вылизывать пиксели — строить систему, в которой правки дешёвые.
6. **Cut-list.** Ядро (разделы 5–10) обязательно к 25 июня. Модули раздела 11 строятся только при остатке времени, в указанном порядке приоритета.

## 1. Что это за продукт

Сервис анализа детских рисунков для русскоязычных родителей. Родитель загружает 1–3 рисунка ребёнка с контекстом (возраст, материалы, что было задано), оплачивает, и получает профессиональный PDF-отчёт о развитии: 8 направлений с оценками 1–10, объяснения с привязкой к деталям рисунка, ссылки на исследования (Пиаже, Ловенфельд, Выготский и др.), практические занятия для родителей.

**Позиционирование: образовательное наблюдение, НЕ медицинская/психологическая диагностика.** Дисклеймер обязателен на лендинге, в отчёте и в письмах. Запрещены формулировки, утверждающие чтение эмоционального/психологического состояния ребёнка по визуальным признакам.

Продукт MVP: **Snapshot-отчёт** (1–3 рисунка одного периода). Цены в рублях — конфигурируемые placeholder-значения: 1 рисунок 990 ₽, 2 рисунка 1490 ₽, 3 рисунка 1890 ₽.

## 2. Стек и инфраструктура (решено, не менять)

| Слой | Решение |
|---|---|
| Сервер | Российский VPS, российский домен golosrisunka.ru |
| Приложение | Python, один процесс. Flask или FastAPI — на выбор разработчика |
| БД | SQLite, один файл. Изображения и PDF — на файловой системе, в БД только пути |
| Шаблоны | Jinja2 (страницы сайта И HTML-отчёты — один движок) |
| PDF | WeasyPrint (HTML → PDF). Уже используется в других проектах заказчика |
| LLM | Google Gemini API (vision). Ключ в env |
| Оплата | ЮKassa (hosted checkout + webhook). Фискализация — вне скоупа, решается заказчиком |
| Email | Unisender (transactional API) |
| Фронтенд | Серверный HTML от Jinja2 + глобальный CSS. Без JS-фреймворков и сборщиков. Ванильный JS точечно (загрузка файлов, переключения) |

**Никакого Node.js, Vercel, Supabase, Docker-оркестрации.** systemd-сервис + nginx достаточно.

## 3. Глобальная дизайн-система (обязательное требование)

### 3.1 Принцип
Один файл дизайн-токенов (CSS-переменные) + один файл компонентных классов. Все страницы — лендинг, кабинет, блог, HTML-отчёт, письма (насколько возможно) — используют только их. Новый блок = композиция существующих компонентов. Если блока не хватает — компонент добавляется в общий файл, а не инлайнится в страницу.

### 3.2 Токены (палитра «Синий» — дефолт)
```css
:root {
  /* Палитра по умолчанию: Синий */
  --bg: #F7F1E3;        /* тёплая бумага — фон */
  --paper: #FFFEF8;     /* карточки, полароиды */
  --ink: #1B2533;       /* основной текст */
  --muted: #5A6273;     /* вторичный текст */
  --soft: #8C93A3;      /* третичный текст */
  --accent: #2456A6;    /* главный акцент, CTA */
  --accent2: #C7762B;   /* второй акцент (оранжевый) */
  --badge: #3E8BC9;     /* бейджи, скотч, заливки баров */
  --alt: #EDE4CE;       /* фон чередующихся секций */
  --shadow: rgba(40,60,100,.16);
}
```
Альтернативные палитры (хранить в том же файле закомментированными или классами на `<html>`, переключение — одной строкой конфига):
- **Терракота**: bg #FBF3E4, ink #2A2118, accent #C75B39, accent2 #5E7D4E, badge #E3A72F, alt #F4E8D2
- **Фиолет**: bg #F9F2E9, ink #2C2138, accent #6D3FA8, accent2 #C2702E, badge #9B59B6, alt #F0E5D8

### 3.3 Типографика
- Заголовки: **Rubik** 800/900 (кириллица), letter-spacing слегка отрицательный
- Текст: **Inter** 400/500/600
- Рукописные акценты (подписи к рисункам, бейджи, заметки): **Caveat** 600/700
- Шрифты — **self-hosted** (woff2 в /static/fonts, @font-face, preload). Не Google Fonts CDN.

### 3.4 Базовые компоненты (минимальный набор)
`btn` (+ `btn--primary` с жёсткой тенью 5px 5px 0 var(--ink), hover-сдвиг), `card`, `polaroid` (фото с «скотчем» ::before, поворотом через модификаторы --tilt-l/--tilt-r и рукописной подписью), `badge` (рукописный бейдж-наклейка с оценкой), `score-bar` (строка: название / бар / число; ширина бара = оценка × 10%), `section` (+ `section--alt`), `carousel` (CSS scroll-snap, без JS), `quote`, `step`. Точный состав и именование — на усмотрение разработчика, принцип переиспользования — обязателен.

### 3.5 Дизайн-направление (зафиксировано мокапами)
Тёплая «домашняя» сцена: детские рисунки как полароиды на скотче, слегка повёрнутые, с рукописными подписями и бейджами-оценками («креативность 8/10») прямо на рисунках. Жирная крупная типографика. Заголовок hero: **«У каждого рисунка есть голос. Услышьте его.»** Референс-файлы мокапов прилагаются к проекту (golosrisunka-hybrid.html, golosrisunka-hybrid-2.html — палитра «Синий»).

## 4. Лендинг

### 4.1 Структура (сверху вниз)

5. **Отзывы**: реальные цитаты из закрытого тестирования (тексты даст заказчик), формат «Имя, мама ребёнка (возраст)»

- Семантика, уникальные title/description на каждую страницу, OpenGraph

## 5. Схема данных (ориентир, поля можно расширять)

SQLite. Ниже — логическая модель; точная DDL на усмотрение разработчика.

- **customers**: id, email (unique), created_at
- **children**: id, customer_id, name, birth_info (как введено), created_at — ребёнок привязан к покупателю; повторная покупка с тем же именем ребёнка переиспользует запись
- **orders**: id, customer_id, child_id, product_code (snap1/snap2/snap3), price_kopecks, coupon_code (nullable), status (created / paid / generating / failed / delivered), yookassa_payment_id, created_at, paid_at
- **drawings**: id, order_id, file_path, context_json (все поля формы по этому рисунку), uploaded_at
- **reports**: id, order_id, html_path, pdf_path, report_json_path (сырой JSON от Gemini — хранить обязательно для отладки и будущего Development Report), public_token (для URL отчёта), generated_at, attempts
- **sessions**: id, customer_id, token (cookie), expires_at (30 дней)
- **login_codes**: id, email, code, expires_at (30 мин), used (bool), requested_at
- **coupons** (схему заложить, модуль может быть отрезан): code, percent_off, multi_use (bool), active, uses_count

Файлы: `/data/drawings/{order_id}/...`, `/data/reports/{order_id}/...`. БД хранит пути.

**Гибкость формы**: список полей контекста — конфиг (Python-структура или JSON): ключ, подпись, тип, обязательность. Рендер формы и валидация — по конфигу. Состав полей на запуск: имя ребёнка, возраст (лет/мес.), дата рисунка, настроение, что было задано/тема, материалы, время рисования, что бросилось в глаза родителю, дополнительный контекст.

## 6. Флоу покупки и генерации

```
Лендинг → выбор продукта (1/2/3 рисунка)
→ Форма: email + поля ребёнка + загрузка рисунков + контекст по каждому [+ купон, если модуль есть]
→ Создание order (status=created), сохранение файлов
→ Redirect на ЮKassa hosted checkout
→ Webhook ЮKassa: оплата подтверждена → status=paid
   → создать/найти customer и child
   → создать session, выдать cookie (30 дней) — пользователь сразу «в кабинете»
   → страница успеха: «Отчёт придёт на почту в течение часа» + ссылка на кабинет
→ Фоновая генерация (см. 7)
→ Доставка (см. 8)
```

Технически фоновая генерация — простейший вариант: поток/таск из процесса приложения или cron-воркер, опрашивающий orders со status=paid. Без Celery/Redis.

**Загрузка файлов**: jpg/png/heic→конвертация, лимит размера (~15MB), ресайз до разумного для Gemini (длинная сторона ~2000px). Валидация на клиенте и сервере.

## 7. Генерация отчёта

### 7.1 Пайплайн
```
Gemini (изображения + контекст + системный промпт)
→ строгий JSON (схема 7.3)
→ валидация JSON (структура, диапазоны оценок)
→ Jinja2-шаблон отчёта → HTML (сохранить, hosted по public_token)
→ WeasyPrint → PDF (сохранить)
→ status=delivered, отправка email
```

### 7.2 Отказоустойчивость
- До **5 попыток** вызова Gemini/валидации с паузами; невалидный JSON = неудачная попытка
- После 5 неудач: status=failed, **email-алерт на захардкоженный адрес заказчика** (env-константа ADMIN_ALERT_EMAIL) с order_id и текстом ошибки
- В кабинете failed-отчёт отображается как «в обработке» (не пугать клиента); починка — ручной перезапуск заказчиком (CLI-скрипт `regenerate_report.py <order_id>` — обязателен)

### 7.3 JSON-контракт отчёта (модель отдаёт данные, шаблон владеет всем видом)
```json
{
  "child": {"name": "...", "age_display": "5 лет 6 мес."},
  "context_summary": "...",
  "introduction": "... (этап развития, ссылки на источники)",
  "dimensions": [
    {
      "key": "creativity",
      "title": "Креативность",
      "score": 8,
      "observation": "привязка к конкретным деталям рисунка",
      "research_note": "общая ссылка на исследование (Boyatzis & Varghese, 1994)",
      "activities": ["занятие 1", "занятие 2"]
    }
    // 8 направлений: креативность, эмоциональная выразительность, уверенность,
    // социальная направленность, организованность, контроль импульсивности,
    // решение задач, логическое мышление (последние два можно объединять при нехватке данных)
  ],
  "recommendations": ["...", "..."],
  "conclusion": "...",
  "insufficient_input": false,
  "insufficient_reason": null
}
```
Если `insufficient_input=true` — отчёт не рендерится; заказчику алерт, клиенту письмо с просьбой о доп. информации (текст письма — шаблон).

### 7.4 Требования к системному промпту (промпт пишется в рамках этого проекта, на русском)
База — существующий custom-GPT промпт заказчика, с обязательными исправлениями:
1. **Без эффекта Барнума.** Каждое наблюдение привязано к конкретной видимой детали рисунка. Неоднозначные детали отмечаются как неоднозначные, а не трактуются в лестную сторону.
2. **Запрет на чтение эмоций.** Описывать, что изображено. Не делать выводов об эмоциональном/психологическом состоянии ребёнка по визуальным признакам. Исследования о цвете/композиции приводить как общие сведения, не как вывод об этом ребёнке.
3. **Обработка недостаточного входа**: если рисунок/контекст не позволяет осмысленный анализ — `insufficient_input=true` + причина, вместо выдуманного отчёта.
4. **Строгий JSON-выход** по схеме 7.3, без markdown-обёрток.
5. Тон: профессиональный, тёплый, конкретный. Оценки объясняются. Сильные стороны и зоны роста — честно.

Тестирование промпта (3–5 реальных рисунков) — обязательный этап до запуска; этим же тестом генерируются образцы для лендинга.

### 7.5 HTML-отчёт
Использует ту же дизайн-систему (токены + компоненты). Структура: обложка (лого, имя, возраст, дата) → детали и контекст → введение → сводный блок оценок (score-bar × 8) → разбор каждого направления (наблюдение, исследование, занятия) → рекомендации → заключение → дисклеймер. Бары и «звёзды» — чистый CSS/SVG от чисел. Вёрстка дружелюбная к WeasyPrint: типографика, простые float/блоки, без сложного flex/grid; постраничные разрывы через CSS page-break. Один тестовый рендер PDF — в первые дни разработки.

## 8. Email (Unisender)

Транзакционные письма (шаблоны Jinja2 → HTML, простая вёрстка):
1. **Подтверждение оплаты** (сразу после webhook): «Заказ принят, отчёт придёт в течение часа» + ссылка на кабинет
2. **Доставка отчёта**: PDF вложением + ссылка на HTML-версию + ссылка на кабинет. **Батчинг**: если по одному заказу несколько отчётов готовы в близком окне — одно письмо
3. **Код входа**: 6-значный код, срок 30 минут
4. **Алерт заказчику** при failed
5. **Недостаточный вход**: просьба о доп. материале

Все письма содержат дисклеймер и контакт поддержки.

## 9. Авторизация и личный кабинет

### 9.1 Модель входа (решено)
- Покупка → аккаунт по email создаётся автоматически → на устройстве покупки сразу активная сессия 30 дней (cookie, httpOnly)
- Повторный вход: email → 6-значный код на почту → ввод → новая сессия 30 дней
- Код: одноразовый, живёт 30 минут, **не более 1 запроса кода на email в 10 минут** (на повторный запрос в окне — сообщение «код уже отправлен, проверьте почту»)
- Никаких паролей. Простейшая защита от перебора кода: N неверных попыток → код аннулируется

### 9.2 Кабинет (одна страница, та же дизайн-система)
- Список заказов: дата, продукт, статус («в обработке» / «готов»)
- По каждому готовому заказу: рисунки (миниатюры → просмотр), ссылка на HTML-отчёт, скачать PDF
- Кнопка выхода
- Ничего больше. Без редактирования профиля, без настроек.

HTML-отчёт доступен по непубличному токен-URL (`/r/{public_token}`) — открывается и из письма без входа. Токен длинный случайный.

## 11. Модули по остаточному времени (в порядке приоритета)

Ядро = разделы 4–10. Только если остаётся время до 25 июня:

1. **Блог**: статьи — markdown-файлы на диске (`/content/blog/*.md` с frontmatter: title, description, date, slug), рендер Jinja2, индекс `/blog`, статьи `/blog/{slug}`, Article schema, попадание в sitemap.
